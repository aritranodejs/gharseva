# 🏠 GharSeva – FINAL PRODUCTION DOCUMENTATION (APP-FIRST, LOCATION-AWARE)

---

## 📌 1. Vision

GharSeva is a dynamic, scalable, app-first home services platform that delivers affordable, reliable services while empowering local workers. The system is designed to match customers with nearby workers efficiently using pincode + real-time location.

---

## 🎯 2. Mission

* Deliver high-quality home services at affordable prices
* Build strong customer trust with reliable fulfillment
* Enable local employment with minimal travel for workers
* Operate a fully digital, app-driven, data-first platform

---

## 🧠 3. Core Product Philosophy

> “100% App-First. Backend-driven. Location-aware. Nothing is hardcoded.”

### Key Principles:

* All bookings happen **only inside the app**
* No manual/offline booking channels
* All services, pricing, and locations are dynamic (DB-driven)
* Pincode controls availability; GPS optimizes assignment
* Frontend renders backend data (no hardcoding)

---

# 🏗️ 4. SYSTEM ARCHITECTURE OVERVIEW

## Tech Stack

* Frontend: React Native (Android-first)
* Backend: Node.js (Microservices)
* Database: MongoDB (GeoJSON for location)
* Realtime: WebSockets (Socket.IO)
* Queue/Cache: Redis (assignment queue, timeouts)

## Core Microservices

* Auth/User Service (OTP login, profiles, addresses)
* Service Management (categories, services – dynamic)
* Package Service (bundles, subscriptions)
* Booking Service (orders, lifecycle)
* Worker Service (skills, availability, location)
* Assignment Service (auto-assign engine)
* Payment Service (COD + online later)
* Notification Service (push/SMS; WhatsApp later only for notifications)

## Pincode-Based (Primary – Day 1)

* User address includes pincode
* Backend validates serviceability
* Only services available for that pincode are returned

* Workers send location every 5–10s when online or on a job

* **Pincode → availability & capacity control**

# ⚙️ 8. CORE BUSINESS LOGIC

## Booking Flow:

1. User logs in (OTP)
2. Selects/creates address (pincode)
3. Check serviceability
4. Fetch dynamic services/packages
5. Select service/package & slot
6. Create booking (status: `searching_worker`)
7. Trigger assignment engine

## Booking Status Lifecycle:

* `pending` → `searching_worker` → `pending_acceptance` → `confirmed`
* `in_progress` → `completed` / `cancelled`

---

# 🤖 9. AUTO-ASSIGNMENT ENGINE (CORE)

## Goals:

* Match **same-pincode** workers
* Minimize travel using **nearest GPS**
* Ensure quick acceptance

## Selection Priority:

1. Same pincode
2. Skill match
3. Availability (isOnline + not overloaded)
4. Nearest distance (geo query)

## Flow:

* Fetch top N (1–3) nearest eligible workers
* Send realtime job request
* Worker has 30–60s to accept
* First accept → assign & confirm
* Others auto-cancel
* If none accept → retry next batch or fail gracefully

## Pseudo Logic:

```
workers = findEligible(pincode, skill)
nearest = sortByDistance(workers, userLocation)
for w in nearest[0..N]:
  sendRequest(w)
wait for accept (timeout 30–60s)
if accepted: confirm
else: retry next batch
```
